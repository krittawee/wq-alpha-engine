# Phase 999.1: Hunt→Bruteforce Template Bridge — Specification

**Created:** 2026-06-16
**Ambiguity score:** 0.20 (gate: ≤ 0.20)
**Requirements:** 6 locked

## Goal

`/hunt --bruteforce-handoff` runs a bounded loop where the LLM emits a parameterized
*template* (expression + slots) instead of individual expressions, the existing AI-free
bruteforce engine sweeps every parameter combination, and the loop generates→sweeps→repeats
until it finds the first alpha that passes all BRAIN IS checks AND the additivity gate (or a
budget/template cap is hit) — spending the LLM only on shape-creation, not enumeration.

## Background

Two alpha-finding paths exist and share zero code (verified: no cross-imports between
`hunt.py`/`researcher.py`/`ideator.py` and `bruteforce.py`/`templates.py`):

- **LLM loop** (`/hunt`): `researcher.py` builds a thesis (archetype + insight); `ideator.py`
  turns it into 4–8 individual FastExpr candidates that go straight to grading. It never emits
  a reusable *template*.
- **AI-free sweep** (`/bruteforce`): `templates.py` holds a **hardcoded** list of 4 templates
  (expression + slots) + `expand_slots` (catalog-grounded via `SELECT DISTINCT id FROM datafields`)
  + `probe_spread_sample`. `bruteforce.bruteforce(client, db_path, delay, quota, probe_size,
  template_names)` sweeps the static list (enumerate → validate → probe/abandon → quota-aware
  ≤3-concurrent bulk-sim → additivity gate → `bruteforce_runs` persistence).

There is no runtime path to feed a *generated* template into the bruteforce engine, and no
registry for dynamically-created templates. Live tests this session confirmed both tools work
at delay-0 but find nothing in tiny runs because they test crowded, obvious shapes with no
breadth. The bridge lets the LLM propose a shape once, then sweep it cheaply and broadly.

## Requirements

1. **Template generator (LLM)**: A generation step emits a parameterized template, not loose expressions.
   - Current: `ideator.py` emits individual FastExpr strings; nothing emits a `{expression, slots, archetype}` template dict.
   - Target: Given a researcher thesis/archetype, the LLM produces a template dict in the same shape `templates.TEMPLATES` uses (expression skeleton with `{slot}` placeholders + `slots` map of literal lists and/or catalog filter dicts + `settings_archetype`).
   - Acceptance: An offline test (mocked LLM) shows the generator returns a dict whose keys match the static template schema and whose `expand_slots` output is non-empty against a fixture catalog.

2. **Template validation + catalog grounding**: Generated templates are proven valid before any sim.
   - Current: Static templates are trusted by construction; no validator runs on a template's structure.
   - Target: Each generated template is catalog-grounded and validated — catalog slots expand only to real `datafields` ids (reuse `expand_slots`), and every enumerated expression passes `validate.validate`. A template that yields zero valid combos is rejected and the loop requests another (counts toward the template cap).
   - Acceptance: Offline test — a template referencing an unknown field/operator yields zero valid combos and is rejected (no sim attempted); a valid template yields ≥1 validated combo.

3. **Template registry (persistence)**: Validated templates are stored for reuse and audit.
   - Current: No registry; templates live only as a hardcoded Python list.
   - Target: A registry table (in `db.py`) persists each validated generated template (source thesis/archetype, expression, slots, created_at, originating hunt run_id) and links to its `bruteforce_runs` rows.
   - Acceptance: Offline test — after a generated template is validated, a registry row exists with the expression + slots round-trippable; the matching `bruteforce_runs` row references it.

4. **Handoff entry point** `/hunt --bruteforce-handoff`: One flag wires generation → sweep.
   - Current: `/hunt` only runs research→ideate→grade→mutate; no flag invokes the bruteforce engine.
   - Target: `--bruteforce-handoff` replaces the ideate-individual-expressions step with: generate template → validate/register → call `bruteforce.bruteforce(...)` on that template. Normal `/hunt` (no flag) is unchanged.
   - Acceptance: Offline test — invoking hunt with the flag calls the template generator + `bruteforce.bruteforce`; invoking without the flag does neither (existing hunt path untouched, all current hunt tests still green).

5. **Bounded generate→sweep→repeat loop**: The loop cannot run unbounded.
   - Current: No such loop exists.
   - Target: The loop is bounded by BOTH a total-sim budget (`--max-sims`, summed across all templates in the run) AND a max-template cap (`--max-templates`). It stops when either bound is hit OR the success target (R6) is met — whichever comes first. Respects the existing ≤3-concurrent cap, single-shot auth (no re-auth in loop), and probe/abandon pruning.
   - Acceptance: Offline test (mocked grade/bruteforce) — loop halts at the sim budget, halts at the template cap, and halts immediately when a success-target alpha is returned; never exceeds either bound.

6. **Success target = passes checks + additive (stop at first, never auto-submit)**: The loop hunts a specific, gated target.
   - Current: bruteforce reports survivors; hunt ranks by Sharpe; neither defines "submittable + additive" as a loop-terminating target.
   - Target: The loop stops at the FIRST alpha that passes all BRAIN IS checks AND the additivity gate (`additivity.confirm_additive`). Result is presented as a ranked candidate for human review — the tool NEVER auto-submits.
   - Acceptance: Offline test — given a mocked stream of results, the loop terminates on the first result flagged check-passing + additive and surfaces it; a live human-verify checkpoint confirms an end-to-end `/hunt --bruteforce-handoff` run performs generate→sweep→stop with a ranked result and no auto-submission.

## Boundaries

**In scope:**
- LLM template generator emitting `{expression, slots, archetype}` dicts
- Template validation + catalog grounding (reuse `expand_slots` / `validate.validate`)
- Template registry table in `db.py` linked to `bruteforce_runs`
- `/hunt --bruteforce-handoff` flag + the bounded generate→sweep→repeat loop
- Stop-at-first success target gated on BRAIN checks + additivity
- Offline test suite + one live human-verify checkpoint (phase-7 style)

**Out of scope:**
- **Auto-submit** — the tool always presents ranked candidates for human review, never submits (project rule; user-locked).

**Assumptions (planner may challenge in discuss-phase):**
- New path is strictly flag-gated; normal `/hunt` behavior is unchanged.
- Reuses `bruteforce.bruteforce()` + `expand_slots` + `additivity.confirm_additive` as-is — only adds generation + registry + handoff glue (no engine rewrite).
- Uses only the already-synced catalog (operators + data-fields); no new BRAIN dataset/operator integration this phase.
- Sweep + mutate-NEAR depth (feeding NEAR survivors back into the editor mutate loop) is NOT required for this phase — generate→sweep→repeat is the loop; per-template mutation can be a later enhancement.

## Constraints

- BRAIN sims ~2 min each, hard cap ≤3 concurrent on one shared cached session; the loop MUST honor this and the total-sim budget.
- Single-shot auth only — the loop must NEVER call `login()` / re-auth inside itself (429 BIOMETRICS_THROTTLED risk). Reuse the cached session like `/bruteforce` and `/hunt` do today.
- Submittability/additivity thresholds read from BRAIN's own `is.checks` and live additivity limit — never hardcoded (e.g. no literal 1.25/0.7).
- Generated-template expressions must be FastExpr using only catalog-verified operators/fields; `--delay` threads through unchanged (delay-0 supported).
- LLM is used only for template/shape creation; the sweep itself makes zero LLM calls (so the sweep stage is runnable when Claude quota is exhausted).

## Acceptance Criteria

- [ ] An LLM step returns a template dict matching the `templates.TEMPLATES` schema; `expand_slots` on it yields ≥1 validated combo (offline, mocked LLM)
- [ ] A template with an unknown operator/field yields zero valid combos and is rejected without any sim attempt
- [ ] Each validated generated template is persisted to a registry table and linked to its `bruteforce_runs` row (round-trippable expression + slots)
- [ ] `/hunt --bruteforce-handoff` invokes generate→validate→`bruteforce.bruteforce`; `/hunt` without the flag does neither and all existing hunt tests stay green
- [ ] The loop halts at the `--max-sims` budget, at the `--max-templates` cap, and immediately on the first success-target alpha — never exceeding either bound (offline, mocked)
- [ ] The loop's terminating alpha passes all BRAIN IS checks AND `additivity.confirm_additive`; the run never auto-submits
- [ ] A live `/hunt --bruteforce-handoff` run completes generate→sweep→stop with a ranked candidate (human-verify checkpoint)
